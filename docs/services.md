# Сервисы в Zapret-Zen

Этот документ описывает, как устроены сервисы в Zapret-Zen, как они применяются к активному runtime и как добавить новый сервис без долгого изучения внутренностей проекта.

## Коротко

Сервисы состоят из нескольких частей:

- `src/zapret_zen/services/service_catalog.py`
  Здесь хранится то, что видит пользователь в интерфейсе: `id`, названия, описания, иконка, акцентный цвет.
- `src/zapret_zen/services/service_rules.py`
  Здесь хранится фактическая логика сервиса: домены, IP/CIDR, `hosts`, дополнительные списки, тестовые цели, дополнительные аргументы `winws`.
- `src/zapret_zen/services/components.py`
  Здесь сервисы встраиваются в активный runtime Zapret во время merge.
- `ui_assets/service_icons/<service>.svg`
  Иконка сервиса для онбординга и страницы сервисов.
- `sample_data/default_services/<service_id>/...`
  Дополнительные файлы сервиса. Это лучший способ хранить большие `ipset`-списки и длинные текстовые листы, которые неудобно держать прямо в Python-коде.

## Как работает merge

Во время подготовки runtime слои применяются в таком порядке:

1. Базовый runtime Zapret.
2. Обычные включённые модификации.
3. Выбранные пользователем сервисы.
4. Пользовательские file overrides.

Это значит:

- сервисы применяются поверх обычных модификаций;
- пользовательские ручные правки всё равно имеют более высокий приоритет.

## Из чего состоит правило сервиса

`ServiceRule` сейчас поддерживает такие поля:

- `list_general`
  Домены, которые добавляются в `lists/list-general.txt`.
- `list_exclude`
  Домены, которые добавляются в `lists/list-exclude.txt`.
- `list_google`
  Домены, которые добавляются в `lists/list-google.txt`.
- `ipset_all`
  IP/CIDR, которые добавляются в `lists/ipset-all.txt`.
- `ipset_exclude`
  IP/CIDR, которые добавляются в `lists/ipset-exclude.txt`.
- `hosts`
  Строки для файла `active_runtime/.service/hosts`.
- `extra_lists`
  Дополнительные `.txt`-списки, которые задаются прямо в коде.
- `extra_list_files`
  Дополнительные `.txt`-списки, которые читаются из файлов проекта. Это предпочтительный вариант для больших списков.
- `winws_args`
  Дополнительные аргументы, которые дописываются к итоговой команде `winws`.
- `test_targets`
  Цели для проверки подключения во время онбординга и автоподбора конфигурации.

## Как правильно добавлять новый сервис

### Вариант 1. Небольшой сервис

Если сервис маленький и не требует огромных списков:

1. Добавьте новый preset в `service_catalog.py`.
2. Добавьте соответствующее правило в `service_rules.py`.
3. Положите иконку в `ui_assets/service_icons`.
4. При необходимости добавьте alias-слова в `MainWindow._service_ids_from_failed_targets()`.
5. Проверьте:
   - отображение в онбординге;
   - отображение на странице сервисов;
   - применение к merged runtime;
   - участие в тестовых целях.

### Вариант 2. Большой сервис

Если сервису нужны большие `ipset` или большие текстовые списки, не храните их в Python.

Создайте папку:

```text
sample_data/default_services/<service_id>/
  lists/
    ipset-all.txt
    list-general.txt
    list-exclude.txt
```

После этого подключите нужные файлы через `extra_list_files` в `service_rules.py`.

Пример:

```python
"clouds": ServiceRule(
    list_general=("cloudfront.net", "amazonaws.com"),
    extra_list_files=(
        ("ipset-all.txt", "sample_data/default_services/clouds/lists/ipset-all.txt"),
    ),
    test_targets=(
        ("Amazon AWS", "https://aws.amazon.com"),
    ),
)
```

Здесь:

- первый элемент кортежа — имя файла внутри merged runtime;
- второй элемент — путь к файлу внутри проекта.

## Минимальные требования к полезному сервису

Сервис не должен быть просто визуальной заглушкой.

Желательно, чтобы сервис делал хотя бы одно из следующего:

- добавлял реальные домены в `list-general`;
- добавлял реальные IP/CIDR в `ipset-all` или `ipset-exclude`;
- подключал дополнительный list-файл;
- изменял поведение `winws` через `winws_args`.

Если сервис ничего не меняет в runtime и влияет только на интерфейс, это placeholder, а не полноценный обход.

Текущее исключение:

- `telegram-desktop`
  Этот сервис не меняет списки Zapret.
  Он управляет `TG WS Proxy` и логикой подключения Telegram Desktop.

## В каком формате просить сервис от другого разработчика

Если другой человек хочет прислать вам новый сервис, удобнее всего просить его в таком виде:

```text
my-service/
  manifest.md
  icon.svg
  lists/
    list-general.txt
    list-exclude.txt
    ipset-all.txt
    ipset-exclude.txt
  notes.txt
```

Где:

- `manifest.md` должен содержать:
  - `service id`;
  - название на русском;
  - название на английском;
  - описание на русском;
  - описание на английском;
  - какие файлы обязательны;
  - какие test targets безопасно использовать;
  - нужны ли дополнительные аргументы `winws`;
- `icon.svg` — иконка сервиса;
- `lists/` — только те списки, которые действительно нужны.

## Правила качества для новых сервисов

При добавлении нового сервиса желательно соблюдать такие правила:

- не копировать случайные публичные blocklist-списки целиком;
- не добавлять рекламные, мусорные, поддельные, typo-домены и зеркала только потому, что в имени встречается нужное слово;
- по возможности использовать официальные домены сервиса, CDN-домены и реальные endpoint-ы;
- если можно использовать доменный suffix, лучше использовать его, а не множество случайных единичных хостов;
- добавлять `ipset` только там, где это реально нужно;
- для `test_targets` использовать стабильные и безопасные цели, а не случайные login-only страницы.

## Где сервис реально применяется в коде

Ключевые места:

- `service_catalog.py`
  Описание сервиса для интерфейса.
- `service_rules.py`
  Правила сервиса.
- `components.py`
  Применение правил в runtime:
  - merge доменов;
  - merge IP/CIDR;
  - merge `hosts`;
  - merge файловых списков;
  - добавление test targets;
  - добавление дополнительных `winws`-аргументов.

## Примеры текущих сервисов

- `cloudflare`
  В основном описан прямо в `service_rules.py`.
- `roblox`
  Описан в коде, включая отдельные CIDR.
- `clouds`
  Смешанный вариант:
  - небольшая курируемая часть описана в коде;
  - большой `ipset` лежит в `sample_data/default_services/clouds/lists/ipset-all.txt`.

## Если позже понадобится полностью data-driven система

Следующий логичный шаг в развитии архитектуры:

1. Ввести формат `service.json`.
2. Загружать сервисы динамически из `sample_data/default_services/*/service.json`.
3. Оставить Python только для merge-логики, валидации и UI-обвязки.

Сейчас это ещё не реализовано.

На данный момент самый быстрый и безопасный путь такой:

1. положить данные сервиса в `sample_data/default_services/<id>/`;
2. добавить preset в `service_catalog.py`;
3. добавить правило в `service_rules.py`.

Такой подход уже позволяет:

- не держать тяжёлые списки в Python;
- удобно расширять сервисы;
- не ломать существующий merge-пайплайн;
- добавлять новые обходы гораздо быстрее, чем раньше.
